# Sprint 2.6 Plan — Stale Booking State After Confirmation

**Date:** 2026-05-27  
**Depends on:** Sprint 2.5 (confirmed merged, c6044e2 + Sprint 2.5 commit)  
**Status:** Planning

---

## 1. Problem Statement

### What the user sees

After submitting a booking and landing on `/confirmation`, the user's BookingContext state still contains the completed booking's data: ferry selections, passenger records, `idempotencyKey`, and `bookingReference`. Sprint 2.5 added a "Start New Booking" / "Back to Home" button pair that explicitly resets state on click — but this is opt-in. Any navigation that bypasses those buttons leaves the stale state intact.

Specifically:
- **Browser back button** from `/confirmation` → `/checkout`: the Order Summary sidebar on checkout reads from BookingContext and displays the old booking's items, price, and passengers as if a new booking is in progress. The `bookingReference` is set, meaning a submit attempt would hit idempotency and silently return the first trip's reference rather than failing loudly or creating a new trip.
- **Direct URL entry** to `/checkout` or `/ferry/passenger-details` in a new tab (within the same sessionStorage lifetime): same stale render.
- **User closes the tab after booking, opens a new one** in the same browser session: sessionStorage persists per-origin per-tab in most browsers, so this scenario is typically safe — but edge cases (session restore, tab duplication) can reproduce it.

### Where the stale state lives

`sessionStorage` under the key `travelbeez-booking`, hydrated into `BookingContext` by `BookingProvider` on mount (`lib/booking-context.tsx` lines 301–418). Because `BookingProvider` is mounted at the locale layout level and is never unmounted during same-session SPA navigation, re-hydration from sessionStorage does not re-run when the user presses back. The stale state is entirely in-memory from the moment the booking was completed.

### When it appears

Exclusively in the same browser session, after a booking is confirmed, via any navigation path other than the two explicit reset buttons added in Sprint 2.5.

## 2. Candidate Solutions

### Option A — Auto-RESET on confirmation mount + booking reference in URL

**Mechanism:**  
The checkout server action (or the confirmation page redirect) appends the booking reference as a query param on success: `/confirmation?ref=TB-26-XXXXX`. On mount, `ConfirmationPage` reads `ref` from `useSearchParams()`, stores it in local `React.useState`, then immediately dispatches `RESET` and calls `clearBookingStorage()`. The page renders exclusively from local state for the remainder of the session; BookingContext is clean before the user can navigate anywhere.

If the user refreshes `/confirmation?ref=TB-26-XXXXX`, the query param is still present but BookingContext is empty — the page can detect this (local state is empty after refresh) and either show the reference from the URL alone (reference + WhatsApp fallback link is enough for the user's next step) or show the "No Booking Found" empty state with a nudge to WhatsApp. The WhatsApp URL does not survive refresh under this model unless it is also passed as a param or re-derived from the reference.

**Files touched:**  
`app/[locale]/checkout/page.tsx` (change redirect target), `app/[locale]/confirmation/page.tsx` (read URL param, reset on mount, remove reliance on BookingContext for display).

**Tradeoffs:**  
+ Proactive: state is clean the moment the user lands on confirmation, before any back-button opportunity.  
+ The confirmation page works correctly on refresh (reference visible from URL).  
- Largest scope of the three options; the confirmation page currently derives all display data (ferry details, passengers, car) from context — migrating to URL param or local snapshot requires a richer `ref` param or a read from the DB on the client side.  
- The WhatsApp payment URL is only available in context; it would need to either be stored alongside the reference or regenerated on the client from the reference.

---

### Option B — Guard: redirect-and-reset if `bookingReference` is already set

**Mechanism:**  
At the top of `CheckoutPage` and `PassengerDetailsPage`, add an effect (after hydration) that checks `state.bookingReference`. If non-empty, call `clearBookingStorage()`, dispatch `RESET`, and `router.replace('/ferry')`. The user is silently dropped back to the start of the funnel rather than seeing stale checkout data.

```
// pseudocode — not final
useEffect(() => {
  if (!isHydrated) return
  if (state.bookingReference) {
    clearBookingStorage()
    dispatch({ type: 'RESET' })
    router.replace('/ferry')
  }
}, [isHydrated])
```

**Files touched:**  
`app/[locale]/checkout/page.tsx`, `app/[locale]/ferry/passenger-details/page.tsx`. No changes to confirmation page or booking-context.

**Tradeoffs:**  
+ Surgical: only the pages where stale state is harmful get guards.  
+ Confirmation page is untouched; the reference-on-refresh behaviour is unchanged.  
- Reactive rather than proactive: there is a render flash (hydration completes → guard fires → redirect) during which the stale Order Summary is briefly visible.  
- Sprint 3 targets both of these files (per-passenger contact fix, legacy field cleanup). Doing 2.6 with this option first means touching the same files twice in consecutive sprints — minor churn risk.  
- Does not guard `/ferry/results`, which also displays ferry context data, though that page's stale state is less harmful (no order summary, no submission path).

---

### Option C — Reset on confirmation mount, snapshot display data in local state

**Mechanism:**  
A middle path between A and B. On `ConfirmationPage` mount (inside the existing `useEffect` that fires confetti), snapshot all display data into a `React.useRef` or `useState` immediately, then call `clearBookingStorage()` + dispatch `RESET`. The page renders from the snapshot for its lifetime; context is clean before the user navigates away.

On refresh, context is empty, local snapshot is gone — the page falls through to the existing "No Booking Found" empty state, which shows a "Search Ferries" link. This is a minor UX degradation (user loses the booking summary on refresh) but is acceptable given: (a) the reference was visible and copyable before refresh, (b) the WhatsApp payment link is the primary next action and is typically tapped before closing the page.

**Files touched:**  
`app/[locale]/confirmation/page.tsx` only. No changes to context, checkout, or passenger-details.

**Tradeoffs:**  
+ Smallest scope; single file.  
+ Proactive like Option A; back button lands on a clean context.  
- Refresh loses the booking summary (only the empty state is shown).  
- Relies on the user copying or acting on the reference before refreshing — acceptable for the current WhatsApp-first payment model, but worth revisiting before Kademe 4 (Viva Wallet) where the confirmation page may need to survive a payment provider redirect.

## 3. Consistency With the items[] / Selector Pattern

All three options are consistent with the items[] pattern — none of them change the shape of BookingState or add new action types. The selector functions (`selectOutboundFerry`, `selectReturnFerry`, `selectCarRental`, `selectTotalPrice`) continue to derive display data from `items[]` exactly as established in Sprint 2.

Option C most naturally extends the Sprint 2.5 change: `ConfirmationPage` already imports all four selectors; snapshotting their output into local state before resetting context is a straightforward extension of what the page already does.

## 5. Recommendation: Option C — Before Sprint 3

**Recommended option: C.**

It eliminates the back-button stale-state problem with the least surface area, stays entirely within the file we already modified in Sprint 2.5, and introduces no new state shape or routing conventions. The refresh-loses-summary tradeoff is real but minor under the current payment model; Option A's URL-param approach is the right long-term solution when Kademe 4 adds a Viva Wallet redirect that must survive a round-trip, and can be revisited then.

**Sequencing: Sprint 2.6 before Sprint 3.**

Sprint 3's stated scope is legacy field cleanup in `lib/booking-context.tsx` and a per-passenger contact fix in `ferry/passenger-details/page.tsx`. Option C touches only `confirmation/page.tsx`, which Sprint 3 does not touch — zero file overlap, zero conflict risk. Doing 2.6 first also means the confirmation page is fully clean-state-aware before Sprint 3 lands, which simplifies manual testing of Sprint 3 (no stale-reference noise when re-testing the checkout flow).

If the team later chooses Option B instead of C, sequencing matters more: 2.6 should still come before Sprint 3 to avoid double-touching checkout and passenger-details, but the reviewer should watch for context guard logic colliding with Sprint 3's contact-field restructuring in passenger-details.

## Plan Addendum — 2026-05-27 Review

**Decision:** Option C with localStorage fallback. Approved for implementation.

### Chosen design

**Render modes.** The component exposes a `ConfirmationMode` discriminated union: `'loading' | 'fresh' | 'refresh' | 'empty'`. Each mode maps to a distinct render path:

| Mode | Trigger | Data source | UI |
|------|---------|-------------|-----|
| `loading` | Before BookingProvider has hydrated | — | `return null` (prevents empty-state flash) |
| `fresh` | Context has `bookingReference` + outbound ferry | In-memory `BookingSnapshot` | Full detail: trip, passengers, contact, totals |
| `refresh` | Context empty; valid localStorage record within TTL | `ConfirmationRecord` (non-PII) | Reduced: reference + WhatsApp CTA + "Trip details have been sent to your email — contact us via WhatsApp" |
| `empty` | Neither context nor localStorage has data | — | Existing "No Booking Found" card |

### PII separation rule

Two data structures are created at mode-detection time and are kept strictly separate:

**`BookingSnapshot` (in-memory only — never serialised to any storage):**
All display data needed for the `fresh` render: `FerryRoute` objects, `Passenger[]`, `contactEmail`, `contactPhone`, `searchParams`, computed totals. This is a plain React state value that lives only for the lifetime of the page mount. It is NOT written to sessionStorage or localStorage.

**`ConfirmationRecord` (localStorage only — non-PII):**
```typescript
interface ConfirmationRecord {
  reference: string    // e.g. "TB-26-M6JEU5"
  whatsappUrl: string  // pre-built wa.me deep link
  timestamp: number    // ms since epoch — used for TTL check
}
```
No passenger names, passport numbers, dates of birth, contact email, contact phone, or ferry details are written here. The `whatsappUrl` may encode the reference number in the URL path but contains no personal data.

### 24-hour TTL

`readConfirmationRecord()` compares `Date.now() - record.timestamp` against `TTL_MS = 24 * 60 * 60 * 1000`. If expired, the entry is deleted and `null` is returned, falling through to the `empty` state. This prevents stale references from surfacing after the booking window has closed.

### Cleanup hook — `handleNewBooking`

The `handleNewBooking(href)` introduced in Sprint 2.5 gains one additional call: `clearConfirmationRecord()`. This ensures that when the user explicitly starts a new booking (either button), both the sessionStorage booking entry and the localStorage confirmation record are cleared before navigation. This prevents the `refresh` view from showing on the next booking's confirmation page if the user completes two bookings in quick succession within 24 hours.

### Hydration timing constraint

`ConfirmationPage` is deeper in the React tree than `BookingProvider`. React runs passive effects (`useEffect`) bottom-up in the component tree, so `ConfirmationPage`'s effect runs **before** `BookingProvider`'s hydration effect. If the mode-detection effect ran on `[]` deps, it would always see empty context and fall through to localStorage/empty, missing the fresh booking.

**Solution:** use `[state.idempotencyKey]` as the effect dependency. `state.idempotencyKey` is `''` in `initialState` and becomes a non-empty UUID only after `BookingProvider` has hydrated (either restoring from sessionStorage or generating a new key for a fresh session). This guarantees the mode-detection effect runs with fully hydrated state.

A `modeDetectedRef = React.useRef(false)` guard prevents the effect from re-running a second time: after a `fresh` detection, the RESET dispatch generates a new UUID, which would change `state.idempotencyKey` again and trigger the effect. The ref short-circuits that second run before it can overwrite `'fresh'` mode with `'empty'`.

### Files changed

`app/[locale]/confirmation/page.tsx` only. No changes to `lib/booking-context.tsx`, `checkout/page.tsx`, or `ferry/passenger-details/page.tsx`.
